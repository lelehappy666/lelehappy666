# GitHub 个人主页 README 美化设计

## 设计目标

把当前以文字和数据表为主的个人主页升级为“沉浸式交互技术作品集”，重点表达 Unity、实时图形、展厅交互、嵌入式设备与网络协议能力，同时保证内容可维护、可访问，并符合 GitHub README 的渲染限制。

视觉设计稿只用于确定视觉层级和氛围，正式页面不得使用一张整页截图代替内容。

## 实现原则

- 页面结构由 Markdown 和 GitHub 支持的 HTML 标签组成。
- 标题、简介、经历、项目名称、技术说明和链接必须是真实文本，可复制、可搜索、可单独编辑。
- 图片只承担视觉素材职责，包括顶部横幅、项目封面和项目演示 GIF。
- GitHub 统计图继续使用各自独立的远程图片，不合并成一张大图。
- 不使用 GitHub 会过滤的 CSS、JavaScript、`iframe` 或脚本动画。
- 仓库内资源使用相对路径，并提供中文或清晰英文替代文本。

## 页面结构

### 1. 顶部横幅

- 使用独立的沉浸式展厅横幅图片，表现粒子轨迹、空间线框、传感器节点与屏幕之间的数据连接。
- 横幅图片不内嵌姓名、职位或介绍文字，避免文字失真，也便于后期替换。
- 横幅下方继续使用 Markdown/HTML 显示 `LE / 乐`、职位、技术方向和联系徽章。
- 页面不模拟 CSS 背景；横幅是正常文档流中的首张通栏图片。

### 2. 个人定位

- 将现有 About Me 压缩为一段定位说明和三项核心能力。
- 三项核心能力使用三列表格展示：沉浸式交互、实时图形、设备与连接。
- 表格内容均为文本和小图标，不制作成图片。

### 3. 技术栈

- 保留现有 Shields.io 徽章。
- 按“实时开发”“视觉渲染”“设备通信”“工具链”分组，减少无序堆叠。
- 不增加与当前经历无关的技术标签。

### 4. 精选作品

- 新增 `Selected Work`，使用两列表格展示重点项目。
- 每个项目单元格由独立封面或 GIF、可点击项目名、技术标签和一句说明构成。
- 优先展示 `SpaceshipDemo` 和 `VisualEffectGraph-Samples`；没有真实演示素材时先使用独立静态封面，不伪造项目运行画面。
- 后续用户提供录屏后，可将对应封面替换为 5～8 秒、无声、循环的优化 GIF，而无需改动 Markdown 结构。

### 5. 仓库与经历

- 精简现有仓库长表格，只保留代表性项目及清晰描述。
- 职业经历继续使用文本表格或时间线式列表，内容不转为图片。

### 6. GitHub 数据

- 保留现有 `2×2` GitHub Overview 布局。
- 保留 Trophies，但降低视觉优先级，放在统计区之后。
- 不新增 Contribution Snake、Spotify 或更多第三方统计组件，避免页面过长和外部服务依赖过多。

## 资源规划

```text
assets/profile/
├── hero-banner.png
├── spaceship-demo-cover.png
└── vfx-samples-cover.png
```

- `hero-banner.png`：通栏横幅，不含文字。
- `spaceship-demo-cover.png`：独立项目封面；未来可替换为同名或新路径 GIF。
- `vfx-samples-cover.png`：独立项目封面；未来可替换为同名或新路径 GIF。
- 当前整页视觉设计稿不复制进仓库，也不在 README 中引用。

## 响应式与兼容性

- 以 GitHub 默认 README 宽度为主要展示环境。
- 使用百分比宽度的图片与简单 HTML 表格，不使用固定页面宽度。
- 横幅、项目封面与统计图片均提供 `alt` 文本。
- 小屏幕上允许 GitHub 对表格整体缩放；避免三列以上的高密度图片布局。

## 验收标准

- README 不是整页图片，页面主要信息均为可编辑文本。
- 顶部横幅、项目封面和统计卡片均为相互独立的图片资源。
- 页面顺序为：横幅与身份信息、个人定位、核心能力、技术栈、精选作品、仓库、经历、GitHub 数据、奖杯与页脚。
- 所有现有有效链接和个人信息保持正确。
- GitHub 不支持的 CSS、JavaScript、`iframe` 和脚本动画均未引入。
- 无真实素材时不伪造项目演示 GIF，保留后续替换接口。
